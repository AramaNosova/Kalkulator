# Регламент процесса разработки проекта "Кондитерский Калькулятор"

## 1. Общие положения

### 1.1. Цель документа
Настоящий документ устанавливает правила, процессы и стандарты работы над проектом "Кондитерский Калькулятор" для обеспечения эффективной коллаборации, прозрачности и достижения поставленных целей.

### 1.2. Команда проекта
- **Разработчик 1:** Анастасия - backend-разработка, архитектура БД, бизнес-логика
- **Разработчик 2:** Александра - frontend-разработка, UI/UX дизайн, тестирование
- **Заказчик:** Алина - постановка задач, приемка результатов, бизнес-экспертиза

### 1.3. Период разработки
18.11.2025 - 30.12.2025 (MVP)

## 2. Организация рабочего процесса

### 2.1. Спринтовая модель разработки
- **Длительность спринта:** 7 календарных дней
- **День начала спринта:** Воскресенье
- **День завершения спринта:** Суббота
- **Количество спринтов:** 6 спринтов до сдачи MVP

### 2.2. Структура спринта

#### 2.2.1. Планирование спринта
- Формирование задач на основе приоритетов заказчика
- Распределение задач между разработчиками по компетенциям

#### 2.2.2. Еженедельная работа
- **Формат:** Асинхронная работа с обновление статуса по готовности задания
- **Коммуникация:** Личный мессенджер для оперативных вопросов
- **Отчетность:** Обмен текстовыми отчетами о прогрессе

#### 2.2.3. Демонстрация результатов (Суббота)
- **Время:** Суббота (гибкое время по согласованию)
- **Длительность:** 30-60
- **Формат:** Чат с отправкой итогов выполнения задачи
- **Участники:** Оба разработчика + заказчик Алина
- **Повестка:**
  1. Обзор выполненных задач за спринт
  2. Демонстрация функциональности
  3. Обратная связь от заказчика
  4. Обсуждение проблем
  5. Предварительное планирование следующего спринта

## 3. Система коммуникации

### 3.1. Каналы коммуникации

#### 3.1.1. Личный диалог разработчиков
- **Назначение:** Оперативное обсуждение технических вопросов
- **Используемый инструмент:** Telegram
- **Правила использования:**
  - Решение текущих проблем разработки
  - Согласование интерфейсов взаимодействия
  - Обмен кодом и скриншотами

#### 3.1.2. Общий канал/репозиторий
- **Назначение:** Хранение и контроль версий файлов проекта
- **Используемый инструмент:** GitHub
- **Правила использования:**
  - Все значимые изменения проходят через pull request

#### 3.1.3. Встречи с заказчиком
- **Назначение:** Демонстрация прогресса и получение обратной связи
- **Формат:** Тестовые встречи с получение обратной связи
- **Частота:** Еженедельно (по субботам)
- **Протокол встречи:** Краткие тезисы сохраняются в общем чате

### 3.2. Правила коммуникации
- **Время ответа:** В любое время

## 4. Управление задачами и отчетность

### 4.1. Система задач
- **Backlog:** Общий список всех задач проекта
- **Спринт бэклог:** Задачи, выбранные для текущего спринта
- **Статусы задач:**
  - Новые задач → В работе → Проверка/Тестирование → Выполненные

### 4.2. Отчетность

#### 4.2.1. Отчетность по готовности (разработчики)
- **Формат:** Текстовое сообщение в личном диалоге
- **Содержание:**
  - Что сделано
  - Время затраченное

#### 4.2.2. Еженедельная отчетность (заказчику)
- **Формат:** Презентация на субботней встрече
- **Содержание:**
  - Обзор выполненных задач за спринт
  - Демонстрация работающей функциональности
  - План на следующий спринт

## 5. Разрешение конфликтов и принятие решений

### 5.1. Принципы разрешения разногласий

#### 5.1.1. Технические решения
- **Метод:** Дискуссионный подход
- **Процесс:**
  1. Каждый участник представляет свое видение с аргументами
  2. Обсуждение плюсов и минусов каждого варианта
  3. Поиск компромиссного решения
  4. Фиксация решения в комментариях к задаче

#### 5.1.2. Продуктовые решения
- **Метод:** Приоритет мнения заказчика
- **Процесс:**
  1. Разработчики предлагают варианты реализации
  2. Заказчик выбирает наиболее соответствующий бизнес-целям
  3. При технической невозможности - совместный поиск альтернативы

#### 5.1.3. Межличностные конфликты
- **Профилактика:**
  - Четкое разделение зон ответственности
  - Уважительное отношение к другому мнению
  - Фокус на решении проблемы, а не на личности
- **Эскалация:** При невозможности разрешения - обращение к заказчику как арбитру

### 5.2. История принятия решений
- Все значимые решения фиксируются в:
  - Комментариях к задачам
  - личном чате/ беседе

## 6. Работа с низкой вовлеченностью участников

### 6.1. Мониторинг вовлеченности
- **Метрики:**
  - Активность в обсуждениях
  - Своевременность выполнения задач
- **Частота проверки:** Еженедельно на субботней встрече

### 6.2. Процесс реагирования

#### 6.2.1. Раннее обнаружение
- Пассивность в обсуждениях

#### 6.2.2. Действия при обнаружении
1. **Персональный запрос:** Вежливый вопрос о причинах через личный канал
2. **Анализ обстоятельств:** Выяснение внешних факторов
3. **Корректировка нагрузки:** Перераспределение задач при необходимости
4. **Прямой разговор:** Обсуждение ситуации на субботней встрече

#### 6.2.3. Профилактические меры
- Четкие и достижимые дедлайны
- Регулярная положительная обратная связь
- Учет индивидуальных предпочтений в распределении задач
- Гибкий график при сохранении результативности

## 7. Заключительные положения

### 7.1. Вступление в силу
Настоящий регламент вступает в силу с момента его утверждения всеми участниками проекта и действует до завершения разработки MVP.

### 7.2. Изменение регламента
Любые изменения в регламент вносятся по согласованию всех участников проекта с обязательным обновлением документа.

### 7.3. Принципы работы вне регламента
В случаях, не описанных в регламенте, участники руководствуются принципами:
1. Взаимного уважения
2. Профессиональной этики
3. Фокуса на результате
4. Прозрачности действий
